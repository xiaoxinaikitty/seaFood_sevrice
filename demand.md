# 基于 Spring Cloud 的海产品服务平台需求文档

## 1. 项目概述

### 1.1 项目名称
基于 Spring Cloud 的海产品服务平台

### 1.2 项目定位
本系统是一套面向个人学习与交流场景的海产品服务平台，目标是完整展示基于 React + Spring Boot + Spring Cloud 的前后端分离微服务系统设计能力。系统不以真实商业交易为目标，而是围绕海产品信息展示、供需信息发布、学习型订单流转、内容交流、后台管理等核心业务进行设计与实现。

### 1.3 建设目标
- 体现 Spring Cloud 微服务架构在用户、商品、订单、内容、消息、权限等领域的拆分方式。
- 完整覆盖前台用户端与后台管理端的主要业务闭环。
- 支持海产品信息展示、供需匹配、模拟下单、平台管理、内容交流等典型场景。
- 在保证系统完整度的前提下，避免引入真实支付、真实物流、复杂结算等超出学习场景的能力。

## 2. 需求澄清与关键结论

### 2.1 场景边界
- 本系统仅用于个人学习、课程设计、毕业设计和技术交流。
- 系统不涉及真实在线支付、资金清算、发票、税务、售后赔付等商业化能力。
- 系统中的“订单”更适合定义为学习型订单、意向订单或模拟交易单，用于体现业务流程，而非真实资金交易。
- 系统需要一次性完整设计与开发，不采用 MVP、分期交付或开发周期规划方式。

### 2.2 角色关系分析
用户提出的核心问题是“普通用户是否需要再细分为卖家和买家”。结合本项目定位，结论如下：

**结论：需要在业务能力上区分买家与卖家，但不建议在账号体系上拆成两套独立用户。**

原因如下：
- 海产品平台天然同时存在“浏览采购”和“发布供应”两类行为，如果强行拆成两套账号，注册、登录、权限、个人中心都会变复杂。
- 本系统是学习型项目，不做真实电商闭环，没有必要设计复杂的商家入驻、企业认证、店铺结算、提现对账体系。
- 很多用户既可能浏览购买，也可能发布供货信息，允许一个账号同时拥有“买家能力”和“卖家能力”更符合实际。
- Spring Cloud 微服务下，推荐使用“统一用户中心 + 角色/身份扩展”的方式处理权限，既体现设计能力，也能控制复杂度。

### 2.3 推荐角色模型
建议采用“1 套账号体系 + 多业务身份”的设计：

| 角色 | 定位 | 核心说明 |
| --- | --- | --- |
| 游客 | 未登录访问者 | 可浏览首页、商品信息、文章内容、公告等公开信息 |
| 普通用户 | 平台基础用户 | 可注册登录、浏览商品、收藏、评论、发布求购、提交模拟订单 |
| 供应者 | 具备卖家能力的用户 | 在普通用户基础上，新增发布海产品供应信息、维护商品、处理订单等权限 |
| 管理员 | 后台运营与管理人员 | 负责用户审核、商品审核、内容管理、订单监管、数据统计、公告发布 |

进一步说明：
- “买家”不单独作为独立账号类型，而是普通用户默认具备的业务身份。
- “卖家”建议命名为“供应者”或“供应方”，作为普通用户的扩展身份存在。
- 一个账号可以同时是“普通用户 + 供应者”。
- 管理员与前台用户账号体系可以统一存储，但权限域必须分离。

### 2.4 产品形态结论
系统建议由两部分前端组成：
- 用户端：面向游客、普通用户、供应者，技术栈采用 React。
- 管理端：面向管理员，建议后续单独建设 React 管理后台，或在同一前端工程中按路由进行后台模块隔离。

## 3. 产品目标与范围

### 3.1 产品目标
- 为用户提供海产品浏览、分类筛选、详情查看、供需交流、学习型下单等能力。
- 为供应者提供海产品信息发布、供货信息管理、订单处理、数据查看等能力。
- 为管理员提供用户、商品、内容、订单、公告、统计等后台管理能力。
- 通过微服务体现认证鉴权、业务拆分、服务治理和接口协作能力。

### 3.2 系统范围
本次完整版本建议包含以下业务域：
- 用户与权限
- 海产品商品中心
- 供需信息中心
- 模拟订单中心
- 内容资讯中心
- 消息通知中心
- 后台运营管理中心

### 3.3 不纳入本次范围的能力
- 真实在线支付
- 真实物流轨迹对接
- 电子发票
- 供应链金融
- 多商家结算与分账
- 小程序、App、多端同步开发

## 4. 用户角色与权限设计

### 4.1 游客
- 浏览首页、商品列表、商品详情、资讯文章、平台公告。
- 使用搜索、分类筛选、查看推荐内容。
- 不可发布信息、下单、评论、收藏。

### 4.2 普通用户
- 注册、登录、退出登录、修改个人资料。
- 浏览商品、加入收藏、发表评论、查看资讯。
- 发布求购信息。
- 提交模拟订单或采购意向单。
- 查看订单状态、消息通知、个人行为记录。

### 4.3 供应者
- 继承普通用户全部权限。
- 申请成为供应者，并提交基础资料。
- 发布、编辑、上下架海产品信息。
- 发布供应信息。
- 查看并处理用户提交的模拟订单。
- 查看商品数据、订单数据、评价反馈。

### 4.4 管理员
- 管理用户信息、供应者申请、角色分配、账号状态。
- 管理商品分类、商品信息、审核状态、推荐位。
- 管理供需信息、评论内容、文章资讯、轮播图、公告。
- 监管订单流程、查看统计报表、记录操作日志。

### 4.5 权限设计原则
- 基于 RBAC 进行权限控制，角色与菜单、接口权限绑定。
- 认证与授权集中处理，避免各业务服务重复维护权限逻辑。
- 前台权限与后台权限分域控制，防止普通用户访问后台接口。

## 5. 核心业务场景

### 5.1 商品浏览与采购意向
游客或普通用户进入平台后，可以按分类、产地、价格区间、库存状态等条件浏览海产品信息，查看商品详情后收藏或提交采购意向单。

### 5.2 供应信息发布
具备供应者身份的用户可以发布海产品供应信息，包括名称、分类、规格、价格、图片、库存、产地、简介等，等待管理员审核后展示。

### 5.3 求购信息发布
普通用户可以发布求购需求，说明品类、数量、预算、期望产地、联系方式等，由供应者查看并进行响应。

### 5.4 学习型订单流转
用户可从商品详情页发起模拟下单，系统生成订单并经历“待确认、待发货、待收货、已完成、已取消”等状态，用于体现订单业务流程。

### 5.5 平台内容交流
平台支持发布海产品科普、饮食知识、养殖常识、保鲜知识等文章，用户可以浏览、点赞、评论，增强平台内容属性。

### 5.6 后台监管
管理员负责审核供应者身份、审核商品、处理违规内容、维护公告和轮播图、查看平台整体运行数据。

## 7. 核心业务流程设计

### 7.1 用户注册与登录流程
游客注册账号后成为普通用户，登录成功后获得前台访问令牌，可访问个人中心、订单、发布求购等能力。

### 7.2 供应者申请流程
普通用户提交供应者申请资料，管理员审核通过后，系统为该用户增加供应者角色，用户即可发布商品与供应信息。

### 7.3 商品发布流程
供应者填写商品信息并提交，系统保存为待审核状态，管理员审核通过后商品上架展示，审核驳回则返回原因供供应者修改。

### 7.4 模拟订单流程
普通用户浏览商品后提交订单，订单进入待确认状态；供应者确认后可标记发货；用户确认收货后订单完成；任一方在规则允许范围内可取消订单。

### 7.5 求购响应流程
普通用户发布求购信息后，供应者可查看并进行响应，平台可通过站内消息通知双方。

## 8. 微服务架构设计建议

### 8.1 总体原则
- 使用 Spring Cloud 体现服务注册、配置管理、网关路由、服务调用、权限控制等核心能力。
- 服务拆分要适度，避免为了“微服务而微服务”导致学习项目复杂失控。
- 推荐按照业务域拆分，而不是按页面拆分。

### 8.2 推荐技术栈
- 前端：React、Vite、React Router、Axios、Ant Design 或 Arco Design
- 后端：Spring Boot、Spring Cloud、Spring Security、JWT
- 服务治理：Spring Cloud Gateway、Nacos
- 服务调用：OpenFeign
- 缓存：Redis
- 数据库：MySQL
- 文件存储：本地存储或 MinIO
- 消息机制：RabbitMQ 或 RocketMQ（可选，用于通知异步化）

### 8.3 推荐微服务划分

#### 8.3.1 gateway-service
- 系统统一入口
- 路由转发
- Token 校验
- 权限拦截
- 限流与统一异常处理

#### 8.3.2 auth-service
- 用户登录认证
- Token 签发与刷新
- 登录安全控制
- 管理员与前台用户认证区分

#### 8.3.3 user-service
- 用户基本信息
- 角色信息
- 供应者申请资料
- 地址簿、收藏、个人资料

#### 8.3.4 product-service
- 海产品分类
- 海产品商品信息
- 商品详情
- 商品上下架
- 商品审核状态

#### 8.3.5 market-service
- 供应信息发布
- 求购信息发布
- 供需广场展示
- 浏览量、响应量统计

#### 8.3.6 order-service
- 模拟订单创建
- 订单状态流转
- 订单明细管理
- 订单取消与完成

#### 8.3.7 content-service
- 文章资讯
- 评论管理
- 公告管理
- 轮播图管理

#### 8.3.8 message-service
- 系统通知
- 审核通知
- 订单通知
- 互动消息

### 8.4 角色关系在微服务中的处理方式
建议采用以下处理模式：

- 用户身份统一存放在 `user-service`。
- 登录认证由 `auth-service` 处理。
- 用户登录后，令牌中携带用户编号、角色标识、权限范围等基础信息。
- `gateway-service` 负责统一校验令牌并将用户上下文透传至下游服务。
- 各业务服务只关心“当前用户是否具备对应角色或权限”，不重复实现登录逻辑。
- 供应者不是独立账号表，而是用户表上的角色扩展，或者通过单独的供应者资料表与用户表一对一关联实现。

推荐角色标识：
- `ROLE_USER`：普通用户
- `ROLE_SUPPLIER`：供应者
- `ROLE_ADMIN`：管理员

这样设计的优势：
- 一个账号可以兼容多角色。
- 权限模型清晰，便于前后端统一控制。
- 微服务之间职责清楚，便于后续扩展。

## 11. 非功能需求

### 11.1 安全性
- 登录接口需进行密码加密存储。
- 管理端接口必须进行严格权限校验。
- 关键操作需记录日志。
- 防止未授权访问后台路由与接口。

### 11.2 可用性
- 页面结构清晰，适合课程演示与论文答辩展示。
- 用户端与管理端操作路径明确。
- 错误提示、状态提示、空数据提示完整。

### 11.3 性能要求
- 常规列表接口支持分页查询。
- 热门数据可基于 Redis 做缓存。
- 图片资源应进行统一管理，避免前端静态资源过大。

### 11.4 可维护性
- 前后端按模块划分清晰。
- 微服务命名统一，接口风格统一。
- 使用统一返回结构、统一异常处理、统一权限模型。

## 12. 最终产品定义

本系统最终应被定义为：

**一套基于 React + Spring Boot + Spring Cloud 的海产品服务平台学习型完整系统，包含用户端、供应者能力、后台管理端三类视角，采用统一账号体系与角色扩展机制，实现海产品信息展示、供需发布、模拟订单、内容交流和后台运营管理。**

关于“用户是否需要分为卖家和买家”的最终结论如下：

**需要在业务权限上区分，但不建议在账号体系上彻底拆分。最佳方案是：普通用户默认具备买家能力，申请通过后扩展为供应者角色，从而兼容买卖双重身份。**

该方案最适合本项目的学习型定位，也最适合 Spring Cloud 微服务架构下的权限与服务设计。
